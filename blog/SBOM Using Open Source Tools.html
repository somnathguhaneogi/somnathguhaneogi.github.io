Software Bill of Materials Using Open Source Tools

A practical guide to generating, managing, and integrating SBOMs into your CI/CD pipeline — without spending a dollar on commercial tooling.
📅 May 2025 ⏱️ 15 min read DevSecOps Supply Chain Security

In the wake of high-profile supply chain attacks, regulators and enterprises alike now expect software producers to provide a detailed list of ingredients in every release. Enter the Software Bill of Materials — and an ecosystem of powerful open source tools ready to generate one.

The term SBOM (pronounced "S-bomb") refers to a structured, machine-readable inventory of all the components, libraries, and dependencies that make up a piece of software. Think of it as a nutritional label for your application: it tells consumers exactly what's inside, where it came from, and whether any known vulnerabilities are lurking in the mix.

With the U.S. Executive Order 14028 on Improving the Nation's Cybersecurity and frameworks like NTIA minimum elements now shaping procurement requirements, generating an SBOM is rapidly shifting from optional best-practice to contractual obligation. The good news? The open source ecosystem has kept pace.

Why SBOMs matter

Modern applications are largely composed of open source packages. A typical Node.js web application might have a package-lock.json with hundreds of transitive dependencies — libraries your code depends on, that depend on other libraries, that depend on yet more libraries. When a critical CVE is published (think Log4Shell, or XZ Utils), you need to know immediately: am I affected?

Key insight: Without an SBOM, answering "do we use Log4j anywhere?" requires manually auditing every codebase. With an SBOM in a centralised store, it becomes a five-second query.

SBOMs enable three critical capabilities:

  • Vulnerability response — rapidly identify which of your products are impacted when a new CVE drops.
  • License compliance — ensure no copyleft-licensed component sneaks into a proprietary product.
  • Provenance attestation — prove to customers and auditors exactly what software you shipped.

SBOM formats: CycloneDX vs SPDX

Before reaching for a tool, you need to decide on a format. Two standards dominate the landscape:

For most teams, CycloneDX is the pragmatic starting point — it has broader tool support and native VEX (Vulnerability Exploitability eXchange) integration. If you are targeting government or pharmaceutical supply chains, consider generating both formats or defaulting to SPDX.

Generating your first SBOM with Syft

Syft is arguably the fastest path from zero to a valid SBOM. It can analyze a local directory, a container image, an OCI archive, or even a running container — and emit CycloneDX or SPDX output in seconds.

Installation

# curl installer (official recommended method)
curl -sSfL https://raw.githubusercontent.com/anchore/syft/main/install.sh | sh -s -- -b /usr/local/bin

# Homebrew (macOS)
brew install syft

# Verify
syft version

Scanning a container image

# Output CycloneDX JSON to stdout
syft nginx:latest -o cyclonedx-json

# Write to a file (recommended for CI)
syft myapp:v2.1.0 -o cyclonedx-json=sbom.cdx.json -o spdx-json=sbom.spdx.json

# Scan a local project directory
syft dir:./my-app -o cyclonedx-json=sbom.cdx.json

Scanning with cdxgen (source-level)

While Syft excels at container image analysis, cdxgen performs deeper source-level analysis, resolving multi-module builds and lockfile-precise dependency trees — particularly valuable for Java, Gradle, and Maven projects.

# Install via npm
npm install -g @cyclonedx/cdxgen

# Generate for a Node.js project
cdxgen -t nodejs -o bom.json ./my-node-app

# Java (Maven)
cdxgen -t maven -o bom.json ./my-java-app

# Python
cdxgen -t python -o bom.json ./my-python-app

# Go module
cdxgen -t go --deep -o bom.json ./my-go-app

Vulnerability scanning with Grype

Once you have an SBOM, feeding it to Grype gives you a vulnerability report in seconds. Grype queries against a locally cached database aggregated from NVD, GitHub Advisory Database, and ecosystem-specific advisories (npm, Maven Central, PyPI, etc.).

# Scan an SBOM file directly
grype sbom:sbom.cdx.json

# Fail the build on high+ severity (useful in CI)
grype sbom:sbom.cdx.json --fail-on high

# Output structured JSON for downstream processing
grype sbom:sbom.cdx.json -o json > vuln-report.json

# Scan an image directly (Syft runs internally)
grype myapp:v2.1.0

CI gate tip: Use --fail-on high as a soft gate in pull request checks. Reserve --fail-on medium for release pipelines, where you have time to triage false positives before shipping.

CI/CD integration: GitHub Actions end-to-end

Below is a production-grade GitHub Actions workflow that generates an SBOM on every push, scans it for vulnerabilities, uploads the SBOM as a build artifact, and optionally submits it to Dependency-Track for continuous monitoring.

.github/workflows/sbom.yml
name: SBOM Generation & Scan

on:
push:
branches: [main, release/*]
pull_request:

jobs:
sbom:
runs-on: ubuntu-latest
permissions:
contents: read
id-token: write # needed for Cosign/OIDC signing
steps:
- uses: actions/checkout@v4
- name: Generate SBOM (Syft)
uses: anchore/sbom-action@v0
with:
path: .
format: cyclonedx-json
artifact-name: sbom.cdx.json
- name: Scan for vulnerabilities (Grype)
uses: anchore/scan-action@v3
with:
sbom: sbom.cdx.json
fail-build: true
severity-cutoff: high
- name: Upload SBOM to Dependency-Track
if: github.ref == 'refs/heads/main'
env:
DT_URL: ${{ secrets.DEPENDENCYTRACK_URL }}
DT_KEY: ${{ secrets.DEPENDENCYTRACK_API_KEY }}
run: |
curl -s -X POST "$DT_URL/api/v1/bom" \
-H "X-Api-Key: $DT_KEY" \
-H "Content-Type: multipart/form-data" \
-F "autoCreate=true" \
-F "projectName=my-app" \
-F "projectVersion=${{ github.sha }}" \
-F "bom=@sbom.cdx.json"

The pipeline in plain English: Syft scans the repository and produces a sbom.cdx.json. Grype reads it and exits with a non-zero code if any high-severity CVE is found — breaking the build. On merge to main, the SBOM is shipped to a self-hosted Dependency-Track instance, where it joins the portfolio and gets re-evaluated every time a new advisory is published.

Continuous monitoring with OWASP Dependency-Track

Generating an SBOM at build time is only half the story. A component that was clean on Tuesday might have a critical CVE published on Wednesday. OWASP Dependency-Track solves this by continuously polling vulnerability databases and alerting when any component in your SBOM portfolio becomes affected.

Spinning up a local instance with Docker Compose takes under five minutes:

docker-compose.yml --- Dependency-Track
version: '3.8'
services:
dtrack-apiserver:
image: dependencytrack/apiserver:4.10
environment:
- ALPINE_DATABASE_MODE=embedded
volumes:
- dependency-track:/data
ports:
- "8080:8080"
dtrack-frontend:
image: dependencytrack/frontend:4.10
environment:
- API_BASE_URL=http://localhost:8080
ports:
- "8081:8080"
volumes:
dependency-track:

After running docker compose up -d, navigate to http://localhost:8081. The default credentials are admin / admin — change these immediately. From there, create a project, generate an API key, and start uploading SBOMs via the workflow shown above.

Signing and attesting SBOMs with Cosign

Generating an SBOM is only valuable if consumers can trust its integrity. Cosign, part of the Sigstore project, lets you sign SBOM files and attach them as attestations to container images — creating a verifiable, tamper-evident chain of custody.

# Attach SBOM as an attestation to an OCI image
cosign attest \
--predicate sbom.cdx.json \
--type cyclonedx \
ghcr.io/myorg/myapp:v2.1.0

# Verify the attestation later (in CI or at deploy time)
cosign verify-attestation \
--type cyclonedx \
--certificate-identity-regexp ".*github.*" \
--certificate-oidc-issuer https://token.actions.githubusercontent.com \
ghcr.io/myorg/myapp:v2.1.0 \
| jq .payload | base64 -d | jq .predicate

Keyless signing in GitHub Actions: Cosign's keyless mode uses the GitHub Actions OIDC token as identity proof. No long-lived secrets, no key management headaches. The signature is anchored in the Rekor transparency log — publicly auditable and immutable.

Common pitfalls and how to avoid them

  • Shallow dependency trees. Tools that only read top-level manifests (package.json, requirements.txt) miss transitive dependencies. Always use --deep or lockfile-based analysis. Syft's --scope all-layers helps with container images.
  • SBOM drift. An SBOM generated at build time is a snapshot. If you ship once and never update the SBOM, it will diverge from reality as dependencies are patched or updated. Automate regeneration on every release tag.
  • Version imprecision. Package managers sometimes resolve version ranges differently across environments. Use lockfiles (package-lock.json, poetry.lock, go.sum) as the source of truth for SBOM generation to guarantee exact version pinning.
  • Ignoring false positives. Not every CVE reported by Grype is exploitable in your context. Establish a VEX workflow: use cdxgen's VEX generation or craft openvex documents to formally record "not affected" assessments, and store them alongside your SBOM.

Putting it all together

An effective open source SBOM program does not require a large budget — it requires discipline. The pattern is straightforward: generate a precise SBOM from lockfiles at build time (Syft or cdxgen), scan it as a CI gate (Grype), sign it for integrity (Cosign), store it with the release artifact, and monitor it continuously against live vulnerability feeds (Dependency-Track).

Start with one project, one format (CycloneDX), and one tool (Syft). Once the pipeline is reliable, scale it across your portfolio. The marginal cost of adding the second project is close to zero — the heavy lifting is in the first one.

Supply chain security is not a destination; it is an ongoing practice. An SBOM is the foundation that makes every other practice — vulnerability management, license compliance, incident response — faster and more accurate.
