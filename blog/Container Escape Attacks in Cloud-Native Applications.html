Container Escape Attacks in Cloud-Native Applications

A practical AppSec research guide explaining how attackers break container isolation in modern cloud-native environments and how penetration testers identify these risks.

Introduction

Modern cloud-native applications rely heavily on container technologies such as Docker and orchestration systems like Kubernetes. Containers provide lightweight environments that allow developers to deploy scalable microservices quickly.

However, containers share the host operating system kernel. This creates a potential security risk known as a container escape attack.

A container escape occurs when an attacker who gains access to a running container is able to bypass isolation controls and execute commands on the host operating system or other containers.

In real-world breaches, container escape attacks often lead to:

  • Host compromise
  • Kubernetes node takeover
  • Cluster lateral movement
  • Credential theft
  • Cloud account compromise

Container Isolation Model

Container security relies on several Linux kernel mechanisms that isolate processes and resources.

  • Namespaces – isolate process views
  • Control Groups – restrict resource usage
  • Seccomp – filter system calls
  • Linux Capabilities – restrict privileged operations
  • Security Modules – AppArmor or SELinux enforcement

If these protections are weakened or misconfigured, attackers may escape the container boundary.

Typical Container Escape Attack Flow

External Attacker | v Web Application Vulnerability | v Remote Shell in Container | v Container Enumeration | v Privilege Escalation | v Container Escape | v Host System Compromise | v Access Other Containers / Cloud Secrets

Exploitation Path 1 — Privileged Containers

One of the most dangerous container configurations is running containers with the privileged flag.

docker run --privileged ubuntu

Privileged containers gain access to all Linux capabilities and host devices. If an attacker gains shell access inside such a container, they may mount host filesystems and modify critical system resources.

Host Filesystem Mount Attack

mkdir /host mount /dev/sda1 /host

This exposes sensitive files such as SSH keys, credentials, and configuration data from the host.

Exploitation Path 2 — Docker Socket Exposure

Another common container escape vulnerability occurs when the Docker daemon socket is mounted inside containers.

-v /var/run/docker.sock:/var/run/docker.sock

This allows containers to communicate with the host Docker engine. Attackers can then control the Docker daemon.

Full Host Escape

docker run -it -v /:/host ubuntu chroot /host bash

This command launches a container with the host filesystem mounted, giving attackers full access to the host environment.

Exploitation Path 3 — Dangerous Linux Capabilities

Containers sometimes run with excessive capabilities such as:

  • CAP_SYS_ADMIN
  • CAP_SYS_MODULE
  • CAP_SYS_PTRACE

These capabilities can allow attackers to interact with kernel resources and mount filesystems.

capsh --print

Exploitation Path 4 — HostPath Volume Abuse

In Kubernetes environments, hostPath volumes allow containers to mount directories from the host.

volumes: - name: hostfs hostPath: path: /

If exploited, attackers may modify host configuration files or implant persistence mechanisms.

Exploitation Path 5 — Kernel Vulnerabilities

Since containers share the host kernel, vulnerabilities in the Linux kernel can lead to container escape.

Well-known vulnerabilities include Dirty COW and Dirty Pipe, which allowed attackers to escalate privileges to root.

Exploitation Path 6 — Kubernetes Privilege Escalation

In Kubernetes environments, attackers may escalate privileges through misconfigured RBAC permissions.

kubectl auth can-i create pods

If allowed, attackers can deploy malicious privileged pods.

Container Enumeration Checklist for Pentesters

After gaining container access, pentesters should enumerate the environment carefully.

cat /proc/1/cgroup mount ps aux ls /var/run/secrets

Practical Container Escape Lab

The following lab demonstrates Docker socket exploitation.

docker run -it -v /var/run/docker.sock:/var/run/docker.sock ubuntu bash
apt update apt install docker.io -y
docker run -it -v /:/host ubuntu chroot /host bash

Defensive Controls

  • Avoid privileged containers
  • Run containers as non-root users
  • Restrict Linux capabilities
  • Never expose the Docker socket
  • Apply Kubernetes Pod Security Standards

Conclusion

Container escape vulnerabilities represent one of the most critical risks in cloud-native infrastructures. Misconfigured containers, dangerous capabilities, and exposed host resources can allow attackers to break isolation boundaries and compromise entire environments.

Security teams and penetration testers should include container escape testing as part of modern application security assessments.
