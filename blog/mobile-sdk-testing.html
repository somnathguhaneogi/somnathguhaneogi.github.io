You audit your own code. But do you audit the code you import? Third-party SDKs collectively represent the largest unreviewed attack surface in most production mobile apps — and attackers know it.

Supply chain attacks against mobile SDKs are rising. In recent years, advertising SDKs have been caught silently harvesting clipboard contents, analytics packages exfiltrated full device fingerprints without disclosure, and payment SDKs shipped with staging credentials hardcoded in production binaries. A single compromised or poorly secured SDK can undo an otherwise solid security posture.

This guide provides a structured, hands-on assessment checklist covering all major risk surfaces: SDK discovery and version auditing, static binary analysis, runtime hooking with Frida, network traffic interception and decryption, data storage inspection, permission analysis, privacy compliance validation, and SDK update mechanism security. Every checklist item maps to the exact tool command you need.

// CAT-01 · SDK INVENTORY
CAT-01
SDK Discovery & Inventory

You cannot assess what you haven't catalogued. Build a complete inventory — including transitive dependencies the dev team may not know exist.

The first step in every SDK assessment is obtaining and decompiling the production binary directly from the app store — not from the development environment. What ships to users and what lives in the repo can differ meaningfully: build scripts, CI configurations, and flavour variants all affect the final binary.

01
Download production binary from the app store
Pull the live production APK/IPA — not a debug build. Use gplaycli (Android) or Apple Configurator 2 (iOS) to obtain the exact binary end users run.
gplaycli / iTunes
02
Decompile APK and enumerate all SDK packages
Use jadx to decompile to Java source. Extract all third-party package namespaces from the class structure. Compare against the dev team's declared dependency list.
jadx / apktool
03
Unpack all bundled .aar and .jar files
SDK A often bundles SDK B which bundles SDK C. Extract nested archives and decompile each to surface the full transitive dependency tree.
unzip + jadx
04
Cross-reference all SDK versions against CVE databases
Run osv-scanner and grype against the decompiled output. Flag any SDK with a known CVE at CVSS ≥ 7.0 for immediate escalation before continuing the assessment.
osv-scanner / grype Critical if CVE
05
Verify SDK vendor ownership and supply chain integrity
Check if SDK package ownership has changed recently — acquisitions, domain takeovers, and maintainer changes are known vectors for supply chain attacks on legitimate SDKs.
Manual OSINT
06
Generate and archive SBOM for the engagement
Produce a Software Bill of Materials (SBOM) in SPDX or CycloneDX format for all discovered SDKs. This becomes the baseline for CVE monitoring and future re-assessments.
syft / cdxgen
cat01-inventory.sh
# ── DECOMPILE & ENUMERATE ─────────────────────────────
$apktool d target.apk -o ./decompiled
$jadx -d ./jadx_out target.apk
# Extract all third-party package roots from smali
$grep -rh "^.class" ./decompiled/smali \
| awk '{print $2}' | cut -d/ -f1-3 | sort -u
# ── CVE SCAN ──────────────────────────────────────────
$grype dir:./jadx_out --output table
$osv-scanner --sbom ./sbom.json

// CAT-02 · STATIC BINARY ANALYSIS
CAT-02Critical
Static Binary Analysis

Reverse-engineer SDK code to expose hardcoded secrets, dangerous API usage, insecure crypto patterns, and privacy-violating data collection — without running a single line of code.

01
Hunt for hardcoded secrets and API credentials
Regex-scan decompiled Java source for API keys, OAuth tokens, AWS credentials, Firebase secrets, private endpoint URLs, and bearer tokens. Run truffleHog with verified-only mode for highest accuracy.
truffleHog / semgrep Critical
02
Identify weak or broken cryptography in SDK code
Search for MD5, SHA-1, DES, AES/ECB mode, hardcoded IV, static seeds passed to SecureRandom, and any custom-rolled crypto implementation. All are exploitable weaknesses.
MobSF / semgrep High
03
Detect dynamic class loading and reflection
Find uses of DexClassLoader, PathClassLoader, Class.forName(), and reflection-based method invocation. These can download and execute arbitrary code at runtime — the core mechanism of SDK update-based attacks.
jadx + grep Critical
04
Scan for device identifier and PII collection patterns
Search for access to ANDROID_ID, IMEI/getDeviceId(), MAC address, GAID/IDFA, clipboard content, contact enumeration, and call log access. Cross-reference with SDK documentation claims.
grep / semgrep High
05
Audit exported Android components registered by SDK
Inspect the merged AndroidManifest for Activities, Services, BroadcastReceivers, and ContentProviders exported by SDK code. Exported components without permission checks are accessible to any app on the device.
aapt2 / MobSF High
06
Check for native (.so) code and JNI boundaries
If the SDK bundles native libraries, static Java analysis is insufficient. Extract .so files and open in Ghidra or Binary Ninja to audit C/C++ code for memory safety issues and hidden functionality.
Ghidra / readelf
Bash — Static Scan static_analysis.sh 
# ── SECRET DETECTION ──────────────────────────────────────────────
trufflehog filesystem ./jadx_out/SDK_PACKAGE --only-verified

# Manual regex sweep for common credential patterns
grep -rEn \
  '(api[_-]?key|secret|token|password|aws_|AIza)["\s:=]+[A-Za-z0-9+/\-_]{20,}' \
  ./jadx_out/ --include="*.java"

# ── CRYPTO WEAKNESS PATTERNS ──────────────────────────────────────
grep -rn \
  "getInstance\(\"MD5\"\)\|getInstance\(\"DES\"\)\|AES/ECB\|SHA-1\|SecureRandom()" \
  ./jadx_out/

# ── DANGEROUS RUNTIME CAPABILITIES ───────────────────────────────
# Dynamic class loading (post-install code execution risk)
grep -rn "DexClassLoader\|PathClassLoader\|loadClass\|forName(" ./jadx_out/

# Device identifier / PII harvesting
grep -rn "ANDROID_ID\|getDeviceId\|getImei\|getMacAddress\
\|getClipboardText\|getContactList\|READ_CALL_LOG" ./jadx_out/

# ── MOBSF AUTOMATED DEEP SCAN ─────────────────────────────────────
curl -X POST -F "file=@target.apk" \
  http://localhost:8000/api/v1/upload \
  -H "Authorization: $MOBSF_API_KEY"
curl http://localhost:8000/api/v1/report_pdf \
  -d "hash=$SCAN_HASH" -H "Authorization: $MOBSF_API_KEY"

// CAT-03 · NETWORK TRAFFIC INTERCEPTION
CAT-03Critical
Network Traffic Interception

Intercept every byte the SDK sends to identify unencrypted channels, undisclosed data exfiltration, missing certificate pinning, and communication with unexpected third-party endpoints.

1
Configure MITM proxy and route device traffic
Set up Burp Suite or mitmproxy. Configure the test device's Wi-Fi proxy to point to your machine. Install the proxy CA certificate on the device under test.
2
Disable SSL certificate pinning at runtime
Production SDKs implement certificate pinning. Use Frida with ssl-kill-switch2 (iOS) or objection's android sslpinning disable command (Android) to strip pinning before interception.
3
Exercise every SDK-powered feature systematically
Trigger: SDK initialisation, analytics events, ad loading, crash reporting, feature flag checks, A/B test evaluation, push notification registration, and user identity operations. Capture all resulting network calls.
4
Map all SDK destination domains
Build a complete domain inventory from captured traffic. Flag: undisclosed domains, data broker domains, ad exchange endpoints, and any endpoints in unexpected geographies. Cross-reference against privacy policy disclosures.
5
Inspect request bodies for PII and sensitive data
For each SDK endpoint, inspect the full request body. Look for: email, phone number, device IMEI, GAID/IDFA, precise GPS coordinates, IP address, clipboard content, and behavioural data. Flag any PII not disclosed to users.
6
Test cleartext fallback behaviour
Force HTTPS to fail (drop connections to SDK domains) and observe if the SDK retries over HTTP. Check Android Network Security Config for cleartextTrafficPermitted="true" exceptions injected by the SDK. Check iOS App Transport Security exceptions.
cat03-network.sh
# ── ANDROID: bypass SSL pinning via objection ─────────────
$objection -g com.target.app explore
> android sslpinning disable

# ── iOS: ssl-kill-switch2 via Frida ───────────────────────

# ── Capture traffic with mitmproxy ────────────────────────
$mitmdump -w sdk_capture.mitm
# ── Filter and extract unique SDK domains ─────────────────

// CAT-04 · DYNAMIC RUNTIME ANALYSIS
CAT-04High
Dynamic Runtime Analysis with Frida

Hook into live SDK execution to observe method calls, intercept data before encryption, modify return values, and reveal runtime behaviours that static analysis cannot surface.

Frida is the most powerful tool in the SDK assessor's arsenal. It lets you inject JavaScript into a running process, hook any method in the JVM or Objective-C runtime, read arguments and return values, and even modify behaviour mid-execution. The most impactful finding class from dynamic analysis is consistently PII collected before transmission — visible only at the hook point, invisible after the SDK's own encryption layer.

Frida — JavaScript sdk_runtime_hooks.js 
// ══ HOOK SDK DATA COLLECTION METHODS ══════════════════════════════
Java.perform(function() {

  // 1. Hook SDK initialization — see what it collects on startup
  const SDK = Java.use("com.analytics.sdk.AnalyticsManager");

  SDK.initialize.overload("android.content.Context", "java.lang.String")
    .implementation = function(ctx, key) {
      console.log("[FRIDA] SDK.initialize() called");
      console.log("  API Key passed in: " + key);
      // Check what context data it harvests during init
      return this.initialize(ctx, key);
    };

  // 2. Hook outbound payload BEFORE SDK encrypts it
  const Collector = Java.use("com.analytics.sdk.DataCollector");

  Collector.buildPayload.overload("java.util.Map")
    .implementation = function(data) {
      console.log("[FRIDA] PRE-ENCRYPTION PAYLOAD:");
      console.log("  " + data.toString());  // full plaintext before encryption
      return this.buildPayload(data);
    };

  // 3. Monitor ALL filesystem writes by the SDK
  const FOS = Java.use("java.io.FileOutputStream");
  FOS.$init.overload("java.lang.String").implementation = function(path) {
    if (path.includes("sdk") || path.includes("analytics")) {
      console.log("[FRIDA] SDK writing to: " + path);
      console.log(Java.use("android.util.Log")
        .getStackTraceString(Java.use("java.lang.Exception").$new()));
    }
    return this.$init(path);
  };

  // 4. Hook clipboard access (common data harvesting path)
  const CM = Java.use("android.content.ClipboardManager");
  CM.getPrimaryClip.implementation = function() {
    console.log("[FRIDA] ⚠️  SDK accessed CLIPBOARD!");
    console.log(Java.use("android.util.Log")
      .getStackTraceString(Java.use("java.lang.Exception").$new()));
    return this.getPrimaryClip();
  };
});

⚠️
The Frida pre-encryption hook is the single most valuable technique in SDK dynamic analysis. It intercepts data in plaintext before the SDK's own encryption layer — revealing the true scope of data collection that network-level analysis alone cannot expose. Always hook buildPayload, serialize, and encrypt method boundaries.
// CAT-05 · DATA STORAGE
CAT-05High
Data Storage Inspection

Examine everything the SDK writes to the device — SQLite databases, SharedPreferences, flat files, and Keychain entries — and verify sensitive data is never stored in plaintext.

01
Pull and inspect all SDK-written SQLite databases
Run the app, trigger all SDK features, then pull the full app data directory. Open every .db file in sqlitebrowser and examine schema and content for PII, session tokens, or behavioural profiles stored in plaintext.
adb pull + sqlitebrowser
02
Inspect SharedPreferences XML files for sensitive SDK data
SDK identifiers, auth tokens, and cached PII are routinely stored in SharedPreferences XML in plaintext. Pull all XML files from /data/data/[package]/shared_prefs/ and grep for credentials.
adb / objection
03
Verify SDK does not write to external (world-readable) storage
External storage is readable by any app holding READ_EXTERNAL_STORAGE. Use a Frida FileOutputStream hook to catch any SDK writes to /sdcard/, /storage/emulated/0/, or other public paths.
Frida hook / adb
04
iOS: Audit SDK Keychain entries and accessibility flags
Dump all keychain items via objection ios keychain dump. Flag items stored with kSecAttrAccessibleAlways — these are accessible even when the device is locked and without biometric confirmation.
objection iOS
05
Check SDK data persistence after user logout
Log out of the app and compare the data directory before and after. SDK data that persists after logout constitutes a privacy violation — especially if it includes identifiers or behavioural profiles.
adb diff

// CAT-06–10 · REMAINING CATEGORIES
CAT-06 → CAT-10
Remaining Assessment Categories

Five additional critical areas required for a complete SDK security assessment.

CategoryKey TestsToolsRisk
CAT-06: Permission Analysis Diff declared permissions in merged manifest vs. runtime usage. Use aapt2 dump permissions to list SDK-injected permissions. Flag ACCESS_FINE_LOCATION, READ_CONTACTS, or READ_CALL_LOG in analytics or advertising SDKs. Strip unnecessary permissions via tools:node="remove". aapt2, MobSF High
CAT-07: SDK Update Mechanism Intercept SDK update channel traffic. Verify updates are signed and signatures are verified before execution. Test if the update endpoint accepts unsigned payloads. A hot-update SDK that doesn't verify signatures is a persistent RCE backdoor. mitmproxy, Frida Critical
CAT-08: Anti-Analysis & Native Code Test if SDK detects Frida, root, or debugger and alters behaviour. Attempt deobfuscation using Simplify or DeGuard. For JNI .so libraries, extract and load in Ghidra for binary analysis. Check for obfuscated URL construction. Ghidra, Simplify Medium
CAT-09: Privacy Compliance Map all collected data against the app's published privacy policy. Test GDPR opt-out and CCPA deletion flows — after opting out, verify the SDK actually stops data transmission via network monitoring. Confirm IDFA/GAID collection requires ATT consent on iOS. mitmproxy, Burp High
CAT-10: SDK-to-SDK Data Sharing Check if SDKs share data via implicit Broadcast Intents, shared SharedPreferences, or ContentProviders. Use logcat monitoring and Frida hooks on sendBroadcast() to detect cross-SDK data flows. Test if one SDK can read another SDK's stored files. Frida, logcat High

ANALYSIS
Secure vs Vulnerable SDK Integration
❌ Vulnerable — SDK relaxes Network Security Config
<!-- SDK merged this into your AndroidManifest -->
<application
 android:usesCleartextTraffic="true">

<!-- network_security_config.xml -->
<base-config>
 <trust-anchors>
  <!-- SDK forced: trust user CAs -->
  <certificates src="user"/>
 </trust-anchors>
</base-config>
<!-- Now any proxy cert is trusted 🚩 -->
✅ Secure — Explicit hardened config
<!-- network_security_config.xml -->
<network-security-config>
 <base-config
  cleartextTrafficPermitted="false">
  <trust-anchors>
   <certificates src="system"/>
  </trust-anchors>
 </base-config>
</network-security-config>
<!-- No user CAs, no HTTP anywhere -->
❌ Vulnerable — Excess permissions from SDK
<!-- SDK injected into merged manifest -->
<uses-permission android:name=
 "android.permission.READ_CONTACTS"/>
<uses-permission android:name=
 "android.permission.ACCESS_FINE_LOCATION"/>
<uses-permission android:name=
 "android.permission.READ_CALL_LOG"/>

<!-- These are for an analytics SDK 🚩 -->
✅ Secure — Strip SDK permission overreach
<!-- Explicitly revoke in app's manifest -->
<uses-permission
 android:name=
 "android.permission.READ_CONTACTS"
 tools:node="remove"/>

<uses-permission
 android:name=
 "android.permission.READ_CALL_LOG"
 tools:node="remove"/>
<!-- Merged build honours these removals -->
// REMEDIATION
REMEDIATION
Remediation Controls
ControlImplementationAddresses
Version pinning & CVE alerting Pin exact SDK versions — never use dynamic ranges (+). Integrate osv-scanner into CI to block builds with known-vulnerable SDK versions. Subscribe to vendor security advisories. Review changelog before any upgrade. CAT-01, CAT-07
Secret rotation and removal Immediately rotate any API key found hardcoded in an SDK binary. Implement server-side key scoping so rotated keys have minimal permissions. Move credentials to server-side proxies — no API keys should ever exist in a mobile binary. CAT-02
Network Security Config hardening Set cleartextTrafficPermitted="false" globally. Trust only system CAs. Never allow user CA trust in production. Audit the merged manifest after each SDK update — SDKs can re-inject relaxations silently. CAT-03
Minimal SDK initialisation Delay all SDK init() calls until user has granted informed consent. Pass only required parameters. Never blindly follow SDK quickstart samples — they maximise data collection, not privacy. CAT-09
Permission stripping After every SDK integration, audit the merged manifest diff. Use tools:node="remove" to explicitly revoke permissions injected by SDKs that your app doesn't need. Document each removal decision. CAT-06
SDK process isolation Run high-risk SDKs (ad networks, analytics) in a separate Android process using android:process. Use IPC boundaries to limit what app data each SDK process can access. CAT-10
Continuous assessment cadence Re-run CAT-01 (inventory + CVE scan) on every SDK version bump. Re-run full assessment (CAT-01 through CAT-10) on major SDK upgrades, after security incidents, and at minimum annually. All
✅
Highest ROI single action: Run MobSF automated scan on every new SDK version before integrating it. Five minutes of automated scanning catches hardcoded secrets, dangerous permission requests, insecure crypto, and exported component vulnerabilities — before they reach production users.
